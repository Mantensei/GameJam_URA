# プロジェクト設計原則

## 1. 子が親に自己登録する（親は子を知らない）

機能を追加するときは、子を足すだけで完了する。親の変更は一切不要。

**Why:** 親が子を監視する設計（UnityのAnimator等）は、機能追加のたびに親を修正する必要があり、拡張性が死ぬ。

**適用方法:**
- 新機能は子として親に自己登録する
- AI/Controllerも下位層に配置し、GameObjectの有効/無効だけで切り替える（「ちぎれる四肢」）
- Hub、Manager等の管理モジュールは例外的に下層を参照してよい

**適用例:** Animator2D + Animation2DRegisterer — 子が親に登録し、優先度ベースで自動解決

## 2. 1モジュール = 機能 + 付随物の自己完結した単位

機能追加はモジュールをぶら下げるだけ。付随物（アニメーション、音等）も同じモジュール内に配置する。付随物を消しても機能本体は影響を受けない。

**Why:** 機能と付随物が別の場所に散らばると、追加・削除時のコストが跳ね上がる。

**具体例:**
```
Player
├── Core/Data          … ステータス（HP, MP等）
├── Art/Visual         … Animator2D + SpriteRenderer（登録を受け付ける側）
├── Component
│   ├── Sensor         … GroundChecker, CeillingChecker
│   └── Action
│       ├── Mover
│       │   ├── Walker … 機能本体
│       │   │   ├── Anim  … 付随物（Animation2DRegisterer）
│       │   │   └── Sound … 付随物（音）
│       │   ├── Flyer
│       │   │   ├── Anim
│       │   │   └── Sound
│       │   └── Runner
│       │       ├── Anim
│       │       └── Sound
│       ├── Jumper
│       │   ├── StandardJumper
│       │   └── MultiJumper
│       └── Misc
│           ├── Reflector
│           └── AntiWallStic
├── Col                … Collider
└── Input
    ├── AI             … ちぎれる。有効/無効で切り替え
    └── Controller     … ちぎれる。AIと排他
```

## 3. データ契約はインターフェースで、クラス継承は実装の便利屋

配列化・差し替えの単位はインターフェースで定義する。クラス継承は共通配線のユーティリティとしてのみ使う。

**Why:** 具象基底クラスに依存すると、インターフェースだけ実装した別の実装を排除してしまう。

**具体例:** 配列にするなら `IMoverProvider[]` であって `MoverBase[]` にはしない。MoverBaseはあくまで共通配線の便利屋。

## 4. 引数が増える可能性があるならパラメータオブジェクトにする

引数を最初からクラスにまとめ、引数追加時の変更範囲の爆発を防ぐ。

**具体例:** MoveCommand — Move(Vector2)等のショートハンドはあくまで窓口。基幹のデータ受け渡しは常にCommandクラス経由。GoFのCommandパターン（実行の遅延/取り消し）ではなく、パラメータオブジェクトとしての実践的な設計判断。

## 5. API利用側はシンプルに使えるようにする

内部がどれだけ複雑でも、利用者にはシンプルなAPIだけを提供する。

**Why:** 利用側が内部の仕組みを理解しなければ使えないAPIは、機能追加のたびに学習コストが発生し、誤用も増える。

**適用例:** 優先度ベースの宣言的マネージャー — Animator2Dでは利用者が触るのはアクティブ/キャンセルだけ。次に何を実行するかは利用者が決めるのではなく、マネージャーが優先度から自動解決する。ステートマシンのような遷移先指定は不要。

**適用例:**
- Animator2D — 複数アニメーションが同時にIsActive=trueでいられ、毎フレーム最高優先度のものが表示される。高優先度が終了すると自然に低優先度に戻る
- InputActionManager — 最高優先度のアクティブなInputMapだけを有効化する。UI(100)が開いている間はPlayer(0)の入力を自動無効化。InputActionPresetが自己登録しSetActive()で宣言するだけ
